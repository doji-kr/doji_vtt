# CLAUDE.md — 화롯가 (Hearthside)

> Synology DS920+에 셀프호스팅하는 5e 호환 TRPG 플랫폼.
> **"TRPG 시나리오의 넷플릭스 + 그것을 만드는 스튜디오 + 상영관(라이브 테이블)을 한 지붕에."**
> DM은 창작자, 시나리오는 작품, 플레이어는 구독자이자 주인공이다.

## 0. 이 문서의 지위

- 이 문서는 프로젝트의 **헌장**이다. 코드와 이 문서가 충돌하면 이 문서가 옳다 — 코드를 고친다.
- §1 절대 원칙의 변경은 반드시 사용자 승인을 받는다. 나머지는 구현이 진행되며 갱신한다.
- 대화로 결정된 사항은 이 문서에 반영한다. **문서에 없으면 아직 결정되지 않은 것이다.**

## 1. 절대 원칙 (The Hearth Rules)

1. **NAS는 릴레이만.** 서버(Celeron J4125, 4코어)는 저장·이벤트 릴레이·시그널링만 한다.
   렌더링은 100% 클라이언트(WebGL), 음성은 WebRTC P2P 메시로 NAS를 거치지 않는다(coturn은 NAT 폴백).
2. **포맷이 심장.** `module.json` 하나를 세 개의 플레이어가 재생한다:
   라이브 테이블(DM 사이드바) · 솔로 러너(게임북) · AI DM(v1.5). — "원소스 멀티플레이"
3. **read_aloud / dm_notes 분리.** 플레이어가 듣는 것과 진행자만 아는 것은 타입 수준에서 분리한다.
   dm_notes는 어떤 코드 경로로도 플레이어 채널에 새지 않는다.
4. **실패도 전진(fail forward).** 모든 판정의 실패 분기는 이야기를 진행시켜야 한다. 린터가 강제한다.
5. **연결의 강약.** soft edge = 라이브 DM이 재량으로 넘나드는 연결, hard edge = 솔로 러너가 따르는 명시 분기.
6. **산수는 기계가, 판정은 사람이.** 주사위 합산·이니셔티브 정렬·HP 기록은 자동.
   성공 선언과 규칙 해석은 DM 몫. 명중→피해 자동 적용 같은 과자동화 금지.
7. **문서 우선 에디터.** DM은 노드 그래프로 생각하지 않는다. 에디터는 글쓰기 + 슬래시 블록이고,
   그래프는 자동 생성되는 조감도일 뿐이다.
8. **픽셀 아트는 공학이다.** 작은 에셋(캠페인당 수십 MB)이 가정용 업로드 대역폭 문제를 해결한다.
   3D·물리 주사위 금지 — 2D 스프라이트 애니메이션(12~16프레임)으로 어디서나 60fps.
9. **단일 커뮤니티 서버.** 동시 라이브 테이블 1~3개 + 솔로 플레이어 다수 규모. 멀티테넌시 없음.
   단순함이 곧 성능이고 온기다. 규모가 커지면 Docker째 VPS로 이사한다.

## 2. 제품 형태 — 세 필러 (v0.5 MVP)

| 필러 | 내용 |
|---|---|
| **플랫폼 셸** | 가입/로그인(아이디+비번, argon2id, 초대코드 게이트 옵션·이메일 불필요), 홈 라이브러리(이어서 하기 / 라이브 / 싱글 서가 / 신작), 이중 프로필(플레이어=모험가 수첩 · DM=이야기꾼의 서재) |
| **라이브 테이블** | 씬(지도 이미지 업로드+그리드 캘리브레이션), 프리셋 픽셀 토큰(종족4×직업6=24종 + 팔레트 스왑), 수동 안개(브러시), /roll 주사위(유리·불리·비밀굴림), IC/OOC/귓속말 채팅, WebRTC 음성(말하는 토큰 글로우), 5e 라이트 시트(클릭=굴림), 이니셔티브, 토큰 HP/상태 |
| **스튜디오 + 솔로 러너** | 문서 우선 에디터(슬래시 블록 5종), 시나리오 린터 + '솔로 재생 가능' 배지, 자동 포스터 생성, 게임북 러너 + 세이브 슬롯(=이어보기) |

**MVP 성공 기준**: "내가 스튜디오에서 만든 시나리오를, 친구는 라이브로 함께 하고,
다른 친구는 혼자 포스터를 골라 완주한다."

## 3. 기술 스택 (고정)

- **Frontend**: TypeScript(strict) + React + PixiJS v8 + Zustand
- **Server**: Node.js 22 + Fastify + ws + better-sqlite3(WAL 모드)
- **검증**: zod — 모든 외부 입력·파일·소켓 경계에서
- **음성**: WebRTC 오디오 메시(≤6인, Opus) + coturn
- **배포**: Docker Compose 2컨테이너(app, coturn) on DS920+ · Synology 역프록시+DDNS 또는 Tailscale
- **테스트**: vitest
- **금지**: Postgres, Redis, Kubernetes, ORM, 마이크로서비스 — J4125에서 전부 오버엔지니어링.
  새 런타임 의존성 추가는 커밋 메시지에 정당화를 쓴다.

## 4. 리포 구조 (pnpm 모노레포)

```
hearthside/
  packages/
    schema/      # module.json 타입·zod·린터·JSON Schema. 의존성: zod만. UI/IO 금지.
    runtime/     # 헤드리스 재생 엔진(상태머신). 의존성 0. 브라우저/노드 양립.
    pixel-ui/    # 디자인 시스템 React 컴포넌트
  apps/
    server/      # Fastify + ws + SQLite + 정적 서빙
    web/         # React + PixiJS 클라이언트
  content/
    modules/     # 손으로 쓴 시나리오 (폴더 = module.json + assets/)
  docker/        # compose, Dockerfile
```

- `schema`와 `runtime`은 **순수 헤드리스 패키지** — 서버·웹·CLI가 같은 코드를 import한다.
  프로젝트에서 테스트 커버리지가 가장 높아야 하는 곳이다.
- 새 기능은 항상 headless 패키지 → 앱 순서로 만든다.

## 5. 데이터 모델 핵심

**module.json** (배포 단위 = 이 파일 + assets/ = zip):
- `meta` — 제목, 로그라인, 포스터, 태그, 난이도, 예상시간, 시작 장면
- `scenes[]` — `read_aloud`, `dm_notes`, `secrets[]`, `blocks[]`, `edges_soft[]`
- `blocks` 5종 — `check`(skill, dc, success/fail goto, fail_forward) · `choice` · `encounter` · `handout` · `secret`
- `npcs[]` — 이름, 초상, **원하는 것 · 두려운 것 · 숨기는 것**(3줄 NPC), 말투 메모
- `flags` — 분기 상태 (예: `장부_위치_안다`)

**SQLite 테이블**: users, profiles, characters, scenarios(발행된 module), campaigns,
sessions(라이브), plays(솔로 세이브), assets

**세이브 = 입력 로그(event-sourced) + 스냅샷.** 같은 입력 시퀀스는 같은 상태(결정성).
라이브 방 상태는 메모리 + 2~5초 디바운스로 SQLite 스냅샷 — NAS 재부팅에도 이어하기.

## 6. 디자인 시스템 「화롯가」

**컨셉**: "친구들이 화롯불 곁에 둘러앉아 이야기를 나누는 밤."
**필라**: 촛불의 온기(어두운 원목 위의 빛) · 손맛(양피지·나무·물성) · 조용한 마법(과하지 않은 인터랙션)

```css
--char: #1B1410;        /* 숯 배경 */        --wood: #2A211A;   /* 목재 패널 */
--wood-edge: #4C3A27;   /* 패널 테두리 */     --parchment: #EFDFB8;
--ink: #3B2D1F;         --candle: #E8A13D;  /* 주 강조 */
--candle-bright: #F4BE6C; /* 호버 */         --ember: #C75146;  /* 위험·HP */
--sage: #8FA872;        /* 성공·회복 */       --moon: #7FA3B8;   /* 정보·마법 */
```

- **타이포**: 제목·라벨 = Galmuri/둥근모 계열 픽셀 폰트(18px 이상, 짧은 텍스트만) · 본문 = Pretendard
- **에셋 규격**: 타일 32px · 토큰 32×32 PNG(인덱스 팔레트 → 팔레트 스왑) · 포스터 400×600(2:3, 없으면 자동 생성)
- **시그니처**: 말하는 사람의 토큰이 촛불처럼 빛난다 · 씬 전환은 촛불 암전 ·
  자연 20 = 금빛 파티클+차임, 자연 1 = 먼지 · 안개는 연기처럼 걷힘 · 사운드는 미세 볼륨, 전부 끌 수 있게
- **의식(ritual)**: 입장 "촛불이 하나 켜집니다" · 세션 시작 = 에피소드 타이틀 카드 ·
  세션 끝 = '오늘의 모험 일지' 리캡 카드(공유 이미지)
- **카피 톤**: 따뜻한 여관 주인. 빈 서가는 "첫 모험의 무대를 펼쳐볼까요?" — 기계적 문구 금지, 이모지 남발 금지.

## 7. 코딩 컨벤션

- TS strict, ESM only. 모든 외부 입력은 zod 파싱 후 사용 — `as` 캐스팅으로 우회 금지.
- 실시간은 서버 권위 상태. 이벤트 명명은 `도메인.동사`: `token.move` `fog.reveal` `dice.roll` `scene.enter`
- 커밋: `feat|fix|chore(scope): 요약` — 한국어 OK.
- 테스트: schema·runtime은 린트 규칙당 위반/통과 픽스처 필수. UI는 핵심 흐름만.
- UI 문자열은 §6 카피 톤을 따른다.

## 8. 성능 예산 (위반 시 머지 금지)

- 세션 중 서버: CPU < 10%, RSS < 500MB (DS920+ 기준)
- 클라이언트: 60fps(중급 노트북), 씬 초기 로드 < 5MB
- 홈 라이브러리: 첫 페인트에 필요한 쿼리 3개 이하

## 9. 범위 밖 (v0.5 Non-goals)

## 11. 법적 가드레일

- 룰 콘텐츠는 **SRD 5.1/5.2 (CC-BY 4.0)** 만 수록. 출처 표기는 `NOTICE.md`에 유지.
- Product Identity(비홀더, 마인드 플레이어 등 고유 크리처명·지명) 사용 금지.
- 대외 표기는 "D&D"가 아니라 **"5e 호환"**.

## 12. Claude Code 작업 수칙

1. 새 기능은 headless 패키지부터. UI는 마지막.
2. **schema를 바꾸면 같은 커밋에서**: 타입 + 린터 + 샘플 모듈 + 러너 테스트를 동시 갱신.
   하나라도 빠지면 그 작업은 미완성이다.
3. 설계가 불확실하면 구현 전에 10줄 이내 설계 요약을 먼저 제시하고 확인받는다.
4. 이 문서와 충돌하는 요청을 받으면 구현하기 전에 지적한다.
5. 성능 예산(§8)을 넘길 수 있는 변경은 측정 결과를 첨부한다.
