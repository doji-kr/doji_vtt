# PROMPT — 4단계: 테이블 완성 (계정 본편 + 5e 라이트 시트·이니셔티브 + 수동 안개 + WebRTC 음성)

> 사용법: 리포 루트에서 Claude Code 새 세션에 이 파일 내용을 그대로 전달한다.
> 전제: 1~3단계 완료 — 테스트 86개 green, 서버 권위 실시간 릴레이(WS)와 PixiJS 라이브 테이블
> (지도·그리드·토큰·핑·채팅·주사위)이 Docker로 NAS에 올릴 수 있는 상태로 동작 중.
> 시뮬레이터 실측 p95 1.75ms, CPU 2.1%, RSS 82.6MB — 3단계 성능 예산 여유 충분.

## 재범위 공지 — 없음, 다만 순서가 중요하다

4단계는 2단계 예고 때부터 이 이름 그대로 예고돼 있던 마지막 항목 묶음이라 추가 재범위는
없다. 다만 "한 단계, 한 리스크" 원칙(3단계 문서 참고)을 이 안에서도 지켜야 한다 — 네 항목의
리스크 성격이 전부 다르다:

| 항목 | 리스크 성격 | 다른 항목과의 결합도 |
|---|---|---|
| 계정 본편(argon2id) | 인증 모델 자체를 바꾼다 — 손대면 세션·소유권·WS identity 전부 영향 | **선행 필수** — 나머지 세 항목이 전부 "누가 이 캐릭터/테이블의 주인인가"를 다시 물어야 하므로 가장 먼저 끝내야 뒤가 편하다 |
| 5e 라이트 시트·이니셔티브·HP/상태 | 새 데이터 모델(캐릭터) + 새 실시간 op — 3단계와 같은 패턴의 확장 | 계정 이후, 독립적 |
| 수동 안개(브러시) | 새 캔버스 레이어 + 새 실시간 op — 3단계와 같은 패턴의 확장 | 계정과 무관, 독립적 |
| WebRTC 음성 + coturn | NAT 통과·브라우저 미디어 정책·mesh 스케일링 — 지금까지와 완전히 다른 리스크 | 가장 고립돼 있다(데이터 모델을 안 건드림) — **맨 마지막**, 시간이 부족하면 여기서 멈춰도 나머지 세 개는 온전히 쓸 수 있다 |

**권장 순서: 계정 본편 → 5e 라이트 시트/이니셔티브 → 수동 안개 → WebRTC 음성.** 이 순서를
따르면 각 단계 끝에서 항상 "완주 가능한 상태"가 유지된다(음성이 없어도 테이블은 완전히
돌아간다). 뒤에서부터 하지 마라.

## 역할과 컨텍스트

너는 「화롯가」의 리드 엔지니어다. 시작 전에 정독하라: `CLAUDE.md`(특히 §1, §2 플랫폼 셸,
§5 데이터 모델, §6 디자인 토큰, §7 이벤트 명명, §8 성능 예산, §9 범위 밖), `docs/STAGE3.md`
(3단계에서 실제로 겪은 함정 두 개 — Pixi `stage.hitArea`, `/assets/` 정적 서빙 충돌 —
비슷한 실수를 반복하지 마라), `docs/PROTOCOL.md`(기존 실시간 프로토콜 v0, 이번 단계에서
확장한다).

**기존 공개 계약은 원칙적으로 변경 금지, 추가만 허용이다** — 단, 계정 본편만은 예외다.
현재 인증은 "닉네임 + 서명 쿠키"뿐이고(`apps/server/src/session.ts`), 이건 진짜 계정이
아니라 자리 표시자였다. 이번 단계에서 이걸 실제 계정으로 바꾸는 것 자체가 목표이므로,
인증 관련 API(`POST /api/session`, `GET /api/session`)와 WS 연결의 identity 판단 방식은
바뀌어도 된다 — 다만 **바꾸는 이유와 새 계약을 `docs/PROTOCOL.md`·`docs/STAGE4.md`에
명시**하고, 기존 REST 리소스(scenarios, plays, tables)의 URL·payload 형태 자체는 유지한다
(주인이 nickname 문자열이 아니라 user id가 되는 것은 내부 구현 변경이지 API 형태 변경이
아니게 설계하라 — 예: 응답 필드명은 그대로 `ownerNickname`으로 유지하되 내부적으로
`users.username`을 가리키게 하는 식). 단일 커뮤니티 서버(§1.9)라 지금 있는 로컬 SQLite
데이터는 마이그레이션 스크립트 없이 갈아엎어도 된다 — 실사용자가 없는 개발 단계다.

## 목표

친구들이 각자 계정으로 로그인해서 자기 캐릭터 시트를 들고 테이블에 앉는다. DM이 안개를
붓으로 걷어가며 던전을 보여주고, 이니셔티브 순서대로 턴이 돌고, HP가 깎이고 상태가 붙는다.
그리고 그 모든 대화가 화면이 아니라 목소리로 오간다 — 브라우저 두 개 이상으로 이 전부가
되면 4단계는 끝난 것이고, v0.5의 "라이브 테이블" 필러가 완성된다.

## 핵심 아키텍처 결정 (구현 전에 이 절을 먼저 제시하고 승인받아라 — 아래 "진행 방식" 참고)

### 1. 계정 — 최소 침습 마이그레이션

- `users(id TEXT PK, username TEXT UNIQUE, password_hash TEXT, display_name TEXT, created_at)`.
  `username`은 로그인 식별자(영문/숫자, 3~20자 정도), `display_name`이 화면에 보이는 이름
  (한글 닉네임 포함, 지금의 `nickname`이 여기로 흡수된다). 비밀번호는 `argon2id`
  (`argon2` 패키지 — CLAUDE.md §2가 이미 알고리즘을 못 박아뒀으니 정당화 불필요, 새 런타임
  의존성이니 커밋 메시지에 한 줄은 남겨라).
- 회원 로그인 세션 쿠키는 `user_id`를 서명해서 담는다(아래에서 보듯 게스트 쿠키는 지금처럼
  `nickname`을 담는 걸 유지 — 완전히 대체되는 게 아니라 새 종류가 하나 추가되는 것이다).
  `requireSession`은 두 종류 쿠키를 구분해서 `request.userId`/`request.displayName`(회원)
  또는 `request.guestName`(게스트)을 채운다. 기존에 `request.nickname`을 쓰던 코드 경로
  (room-registry, table 소유권 비교, WS identity)는 회원/게스트 어느 쪽이든 화면 표시용
  이름 하나로 정규화해서 쓰되(예: `displayName ?? guestName`), **소유권·권한 판단에는 반드시
  `userId`만 쓴다**(게스트는 애초에 `userId`가 없으니 소유권이 필요한 동작에서 자동으로
  걸러진다) — 지금처럼 "닉네임 문자열이 곧 신원"인 구조는 같은 이름을 두 사람이 동시에 못
  쓰게 막는 안전판도 없었다는 걸 상기해라.
- 가입: `POST /api/auth/register { username, password, display_name, invite_code }` —
  기존 사이트 초대코드 게이트(`config.inviteCode`)를 여기로 옮긴다(회원가입 시 1회 확인,
  로그인엔 필요 없음). 로그인: `POST /api/auth/login { username, password }`.
  **`POST /api/session`(닉네임+초대코드, 지금 있는 그대로)은 폐기하지 않는다** — 이게 바로
  아래에서 정의하는 게스트 세션 발급 엔드포인트로 계속 쓰인다.

- **결정됨 — 게스트 입장은 계정 없이 유지, 단 그 자리에서 가입도 할 수 있다**(사용자 확인
  완료). 즉 인증은 두 갈래로 공존한다:
  - **회원 계정** — `users` 테이블, `user_id` 서명 쿠키. 테이블을 **만드는**(DM이 되는) 쪽은
    반드시 이 경로를 탄다("이야기꾼의 서재"는 로그인 전제).
  - **게스트 세션** — 지금의 "닉네임 + 서명 쿠키"를 그대로 둔다(폐기하지 않는다). 초대 링크
    (`/t/:token`)로 들어온 사람은 계정 없이 표시 이름만 정하고 바로 참가할 수 있다. 단
    **사이트 초대코드는 게스트에게도 여전히 요구한다**(회원가입 게이트와 별개로, 지금처럼
    `config.inviteCode`가 "이 서버에 아무나 못 들어오게" 막는 바깥 문 역할을 계속한다).
  - 초대 링크로 들어가는 화면(`JoinByInvite`)에 **"이 이름으로 그냥 들어가기" / "가입하고
    들어가기"** 두 선택지를 둔다. 후자를 고르면 그 자리에서 `POST /api/auth/register`를
    태우고 같은 표시 이름으로 회원 세션을 발급해 테이블에 입장시킨다.
  - **게스트→회원 승격(같은 세션 안에서 매끄럽게 이어가기)은 이번 단계 범위가 아니다.**
    게스트로 한 판 놀고 나중에 따로 가입하는 것과, 지금 당장 "가입하고 들어가기"를 누르는
    것 두 경로만 지원하면 된다 — 이미 게스트로 접속한 세션을 회원 계정으로 승격시키는 별도
    API는 만들지 마라(과설계).
  - `request.userId`/`request.displayName`(회원)과 `request.guestName`(게스트, 지금의
    `request.nickname`을 그대로 가져온 것)을 `requireSession`이 둘 다 채울 수 있게 하고,
    소유권·권한이 필요한 라우트(테이블 생성, 캐릭터 시트 소유 등)는 `userId`가 없으면 403 —
    이게 "게스트는 시트를 못 만든다"의 실제 구현이다. 테이블 **참가**(토큰 이동, 채팅, 공개
    굴림 등 3단계 권한표에 있던 플레이어 권한)는 게스트 세션만으로 충분해야 한다(지금 되던
    걸 유지, 회귀 금지).
  - 캐릭터 시트(`characters` 테이블, 아래 2번)는 `owner_username`이 아니라 `owner_user_id
    NOT NULL`로 둔다 — 즉 **게스트는 캐릭터 시트를 만들 수 없다**(회원가입해야 시트를 가질
    수 있다). 이건 "회원 계정에만 의미 있는 기능"의 첫 사례이니 게스트 UI에서 시트 생성
    버튼 대신 "가입하면 캐릭터를 만들 수 있어요" 안내로 대체해라.
- **이중 프로필의 시작(전부 만들지 마라)**: 계정에 로그인하면 홈 화면에 "모험가 수첩"(내가
  참가 중인 테이블·캐릭터)과 "이야기꾼의 서재"(내가 만든 테이블) 두 진입점만 구분해서
  보여주는 정도로 끝낸다. 완전히 분리된 두 대시보드나 시나리오 발행 흐름은 v0.5 나머지
  범위(스튜디오 에디터)다 — 손대지 마라. 게스트는 이 화면 자체에 도달하지 않는다(로그인
  없이는 홈이 아니라 초대 링크로만 들어오므로).

### 2. 5e 라이트 시트 · 이니셔티브 · HP/상태 — 서버 권위, 최소 필드

- 새 실시간 op(§7 명명 규칙 `도메인.동사`): `character.set`(이름·클래스·능력치 수정치만 —
  전체 스탯블록 아님, "라이트"), `character.hp`(현재/최대 HP 갱신, **델타가 아니라 절대값을
  클라이언트가 보내고 서버가 그대로 반영** — 명중→피해 자동 적용 금지 원칙(§1.6)을 지키려면
  "몇 대 맞았으니 몇 깎였다"를 서버가 계산하면 안 되고, DM/플레이어가 판단해서 숫자를 직접
  써넣는 식이어야 한다), `status.set`(조건 태그 배열 — poisoned/prone 등 자유 텍스트 태그면
  충분하다, SRD 5.1/5.2 조건 목록을 그대로 베끼려 하지 마라), `initiative.set`(참가자별
  순번 숫자 — 굴림은 기존 `dice.roll`을 재사용하고 그 결과를 이니셔티브 값으로 "적용"하는 건
  사람이 클릭해서 확정하는 별도 동작이다).
- 이니셔티브 **정렬**(숫자 비교) 자체는 기계가 해도 된다 — §1.6이 금지하는 건 "판정 해석"이지
  "산수"다. 순번 리본 UI는 정렬된 결과를 보여주기만 하면 된다.
- 데이터 모델: `characters(id, table_id, token_id NULL, owner_user_id NOT NULL, name, class,
  ability_mods_json, hp_current, hp_max, ac, status_json, updated_at)`. `owner_user_id`가
  `NOT NULL`인 게 핵심이다 — 위 1번에서 정했듯 게스트는 시트를 못 만든다. 캐릭터는
  **테이블 범위**로 스코프한다(3단계 토큰과 동일 — 캠페인을 넘나드는 "휴대 가능한 캐릭터"는
  `campaigns` 테이블이 생기는 이후 단계 몫이다, 지금 하지 마라). `token_id`로 토큰과 느슨하게
  연결하되 NULL 허용(시트만 먼저 만들고 토큰은 나중에 놓을 수 있게).
- UI: "클릭 = 굴림" — 능력치 수정치를 클릭하면 그 값으로 `1d20+N` 굴림을 자동 채워서
  `dice.roll` 인풋에 꽂아준다(기존 주사위 파서·채팅 굴림 흐름 재사용, 새 파서 만들지 마라).
  HP 바는 §6 토큰(`--ember`=위험/HP, `--sage`=회복) 색상 규칙을 따른다.

### 3. 수동 안개 — 서버 권위 공유 레이어, 개인별 시야 아님

- CLAUDE.md §9가 "동적 조명/시야"를 명시적으로 범위 밖에 뒀다 — 이건 **플레이어마다 다른
  시야를 자동 계산하는 기능**을 말하는 것이고, 4단계의 "수동 안개"는 그것과 다르다: DM이
  붓으로 지운(reveal) 영역이 **모든 비-DM 참가자에게 공통으로** 보인다(공유 단일 레이어).
  개인별 시야 계산은 하지 마라 — 그건 진짜로 범위 밖이다.
- 새 실시간 op: `fog.init`(그리드 해상도에 맞춘 W×H 비트마스크 초기화, 기본 전체 비공개),
  `fog.reveal`(브러시 스트로크 — 좌표 목록 또는 원형/사각 영역, 서버가 비트마스크에 반영),
  `fog.reset`(전부 다시 가리기). 안개 상태는 `RoomState`에 포함해 스냅샷에 실린다(재접속 시
  복원). DM은 항상 안개 없이 전체를 본다(클라이언트에서 role로 분기 — 서버가 DM 소켓에는
  애초에 안개 마스크 자체를 안 보내는 게 아니라, DM 클라이언트가 안개 레이어를 그리지 않는 것으로
  충분하다 — 이건 비밀 정보가 아니라 그냥 뷰 모드 차이이므로 채널 분리 수준의 엄격함은
  필요 없다).
- 캔버스 레이어 순서 갱신: 지도 → **안개** → 그리드 → 토큰 → 핑(안개가 그리드보다 아래에
  있어야 그리드 선이 항상 보여서 방향감을 유지한다 — 이 순서는 목업이나 실제 플레이 감각을
  보고 조정해도 된다, 강제 아님).

### 4. WebRTC 음성 메시 + coturn — 맨 마지막, 가장 고립된 리스크

- 시그널링은 새 HTTP 서버를 만들지 말고 **기존 테이블 WS 채널을 재사용**한다 — 새 op
  `voice.offer` `voice.answer` `voice.ice`(SDP/ICE candidate를 대상 nickname/userId 지정해서
  중계, 서버는 그냥 릴레이만 하고 내용을 해석하지 않는다). 오디오 자체는 P2P(각 참가자가
  서로 직접 연결하는 완전 mesh, ≤6인 — §3에 이미 못 박힌 제약).
- `docker/docker-compose.yml`에 `coturn` 컨테이너 추가(NAT 통과 실패 시 릴레이 폴백 —
  대부분의 가정용 라우터에선 STUN만으로 충분하니 coturn은 안전망이다). TURN 자격증명은
  서버가 발급(짧은 TTL 공유 비밀 기반 — coturn의 `use-auth-secret` 방식을 써서 DB에 사용자별
  계정을 새로 만들지 마라).
- 브라우저 UI: 말하는 사람 토큰이 촛불처럼 빛나는 연출(§6 시그니처) — 오디오 레벨을
  `AnalyserNode`로 측정해서 임계값 넘으면 해당 토큰에 글로우 클래스를 토글하는 정도로
  충분하다. 음소거 버튼, 전체 꺼짐 옵션(§6 "사운드는 미세 볼륨, 전부 끌 수 있게") 필수.
- **여기서 시간이 부족하면 멈춰라.** 나머지 세 항목이 이미 끝나 있으므로 4단계 부분 완료도
  실사용 가능한 상태다 — 음성 없이 디스코드 병행은 3단계 문서에서도 이미 한 번 쓴 임시책이고,
  여전히 유효한 폴백이다.

## 산출물 (전부 필수, 단 4번 WebRTC는 위 사유로 최선 노력)

1. **계정 본편**: `users` 테이블, argon2id 해시, `/api/auth/register`·`/api/auth/login`,
   기존 nickname 기반 코드 경로 전체를 `userId`/`displayName`으로 이관(room-registry,
   table 소유권, plays 소유권, WS identity), 이중 프로필 진입점 두 개.
2. **5e 라이트 시트·이니셔티브·HP/상태**: `characters` 테이블, 신규 실시간 op 4종, 시트 UI
   (DM·플레이어 모두 자기/소유 캐릭터 편집), 이니셔티브 리본, HP 바, 상태 태그 칩.
3. **수동 안개**: `fog` 상태(RoomState 확장), 신규 실시간 op 3종, 브러시 UI(DM 전용),
   캔버스 안개 레이어.
4. **WebRTC 음성 + coturn**: 시그널링 op 3종, `docker/docker-compose.yml`에 coturn 추가,
   말하는 토큰 글로우, 음소거/전체 꺼짐.
5. **`docs/PROTOCOL.md` 갱신**: 신규 op/이벤트 표, 인증 계약 변경사항, 안개·시트 데이터
   모델을 v0에 이어 v1으로 갱신(버전 표기 추가).
6. **테스트**: 계정(가입/로그인/해시 검증/중복 username 거부), 캐릭터 HP/상태 갱신 e2e,
   안개 reveal이 스냅샷에 반영되는지, 시그널링 릴레이(내용을 안 건드리고 그대로 전달하는지
   — 실제 미디어 연결까지 자동화 테스트할 필요는 없다, WebRTC 연결 자체는 브라우저 필요).
   기존 86개 전부 green 유지가 최우선 — 계정 마이그레이션이 가장 깨뜨리기 쉬운 지점이니
   여기부터 테스트를 다시 통과시켜라.
7. **`docs/STAGE4.md`**: 결정과 근거 / 실제로 검증한 것(3단계처럼 브라우저로 직접 조작 —
   4단계는 계정 로그인 플로우, **게스트 입장과 "가입하고 들어가기" 두 경로 모두**, 이니셔티브
   진행, 안개 걷기, **음성 통화(가능하면)** 까지 사람이/에이전트가 직접 확인해야 진짜로 끝난
   것이다, curl만으로는 부족하다) / 미결·v0.5 나머지 핸드오프.

## 제약

- 새 의존성: `argon2`(또는 `@node-rs/argon2`), WebRTC는 브라우저 네이티브 API라 새 클라이언트
  런타임 의존성 없음(TURN 자격증명 발급용 HMAC 정도는 `node:crypto`로 충분, 새 패키지
  불필요). 그 외는 커밋 메시지에 정당화(§3).
- 성능 예산(§8) 준수 — 안개 비트마스크 크기(그리드 셀 수)가 커지면 스냅샷 페이로드가
  커진다, 씬 초기 로드 5MB 예산을 넘기지 않게 압축(RLE 정도면 충분, 이미지 압축 라이브러리
  끌어오지 마라)을 고려해라.
- 3단계에서 겪은 함정 두 개(Pixi 캔버스 레이어는 반드시 브라우저로 직접 드래그/상호작용까지
  확인, 정적 파일 경로 프리픽스는 겹치지 않는지 빌드 후 실제로 열어봐라)를 반복하지 마라.
