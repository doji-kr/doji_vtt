# 3단계 작업 기록 — 라이브 테이블 코어

> 목표: DM이 방을 열고 지도를 깔고 토큰을 놓으면, 친구가 초대 링크로 들어와 같은 지도 위에서
> 토큰을 움직이고 주사위를 굴린다. 서버 권위 WS 릴레이 + PixiJS 지도·토큰 + 실시간 주사위·채팅.
> DoD는 하단 체크리스트 참고. 재범위 공지대로 WebRTC 음성·coturn·수동 안개·5e 시트·이니셔티브·
> 계정 본편·프리셋 스프라이트·에디터는 전부 4단계로 남겼다.

## 아키텍처 결정

### 서버 권위 실시간 릴레이

`docs/PROTOCOL.md`가 계약의 전문이다. 요약하면: 클라이언트는 `{type, payload}`만 보내는
"제안"이고, `apps/server/src/room-registry.ts`의 `LiveRoom`이 검증(zod, `table-protocol.ts`)·
권한 체크·적용·`seq` 부여·브로드캐스트를 전부 한다. 낙관적 UI는 딱 하나 —
`token.move` 드래그 중 화면 미리보기 — 만 허용되고, 그 외 어떤 값(주사위 결과, 판정 성공 여부
등)도 클라이언트가 계산하지 않는다.

방 상태는 `RoomRegistry`가 테이블 id별로 메모리에 들고 있다가 3초 디바운스로
`tables.state_json`/`grid_json`/`last_seq`에 스냅샷을 저장한다(§5). 재접속은 델타 재생이 아니라
매번 풀 스냅샷을 보내는 방식(v0, PROTOCOL.md에 근거 명시)이라 클라이언트 로직이 단순해진다 —
새로고침이든 첫 입장이든 "연결 → hello → state.snapshot"이 유일한 경로다.

### `@fastify/websocket` 채택 근거

`ws`를 fastify 라우트 시스템에 바로 붙일 수 있고(별도 HTTP 서버 분기가 필요 없다), 쿠키 세션
(`request.cookies`, `preHandler: requireSession`)을 WS 업그레이드 시점에도 그대로 재사용할 수
있어서 골랐다. 대안(`ws`를 fastify 밖에서 직접 붙이는 것)은 세션 미들웨어를 이중으로 관리해야
해서 기각.

**실제로 겪은 함정** — `apps/server/src/app.ts`의 주석에 남겨뒀듯, `@fastify/websocket`을
`await` 없이 `app.register()`만 하면 플러그인의 `onRoute` 훅이 붙기 전에 `.get({websocket:true})`
호출이 동기적으로 먼저 실행돼서, WS 라우트가 조용히 "일반 HTTP 라우트"로 등록돼버린다
(핸들러 인자가 `(socket, request)`가 아니라 `(request, reply)`로 뒤바뀌어 들어와서 에러 없이
그냥 이상하게 동작한다 — 디버깅이 까다로운 부류의 버그). 모든 `app.register()` 호출을
`await`로 순서를 강제해서 해결했다. fastify 플러그인을 여러 개 등록할 때 일반적으로 조심해야
할 패턴이라 다음 단계를 위해 여기 남긴다.

### SPA 경로에 대한 서버 폴백 (이번 단계에서 새로 필요해진 것)

2단계까지는 클라이언트에 실제 URL 라우팅이 없어서 `@fastify/static`의 기본 동작(요청 경로에
해당하는 정적 파일이 있으면 서빙, 없으면 404)만으로 충분했다. 3단계는 초대 링크
(`/t/:token`)와 테이블 화면(`/table/:id`)이 **진짜 URL**이어야 한다(누군가 브라우저 주소창에
직접 붙여넣을 수 있어야 하니까) — 그런데 이 경로들은 실제 파일이 아니라 클라이언트가
`window.location.pathname`을 읽어서 화면을 고르는 가상의 경로다. 그래서 `app.ts`에
`setNotFoundHandler`를 추가해, `/api`·`/ws`·`/content`·`/assets`로 시작하지 않는 GET 요청은
`index.html`을 그대로 돌려주게 했다(진짜 SPA 폴백). API 계열 404는 그대로 JSON 404를 유지한다 —
그렇지 않으면 존재하지 않는 지도 이미지 같은 것도 전부 200 HTML을 반환해버려서 클라이언트가
에러를 감지할 수 없게 된다.

### `GET /api/session` 추가 (whoAmI)

2단계까지 클라이언트는 "유효한 세션이 있는가"만 확인했지(`listMyPlays()`가 401을 던지는지로
판단), "나는 누구인가"를 물어볼 방법이 없었다. 3단계 테이블 화면은 자기 토큰인지 판단해서
드래그 가능 여부를 미리 보여주거나 참가자 목록에서 자신을 표시하는 데 닉네임이 필요해서,
기존 서명 쿠키를 그대로 읽는 조회 전용 라우트 `GET /api/session`을 `routes/session.ts`에
추가했다(새 인증 흐름 아님 — `requireSession` preHandler 재사용). 서버 테스트
(`app.test.ts`)에 로그인 전/후 케이스를 추가했다.

### 라우팅 — 라이브러리 없이 최소한만

CLAUDE.md §3(새 런타임 의존성 최소화)에 따라 라우터 라이브러리는 쓰지 않았다.
`apps/web/src/routing.ts`가 순수 함수 `parsePath(pathname)`로 `/t/:token`, `/table/:id`, 나머지는
`home`으로 매핑하고, `App.tsx`가 부팅 시 한 번 읽어서 시작 화면을 정하고 이후 이동은
`history.pushState`/`replaceState`로 처리한다. 뒤로가기(`popstate`)는 로그인된 상태에서만
최소 지원.

### 비밀 굴림 채널 분리 — 실시간판

서버가 `secret: true` 굴림을 DM 소켓에만 보내는 것은 `room-registry.ts`가 이미 하고 있었다
(2단계 채널 분리 원칙의 확장, `table-ws.test.ts`가 증명). 클라이언트 쪽 원칙은 "받은 걸 그대로
그린다, 숨겨야 하는지 판단하는 코드 경로를 아예 만들지 않는다"다 — `table-reducer.ts`의
`applyServerMessage`는 `dice.roll` payload를 필터링 없이 그대로 log에 쌓기만 한다. 플레이어
소켓에는 애초에 그 이벤트가 도착하지 않으므로 안전.

### 토큰 v0 — 제너레이티브 픽셀 토큰

`apps/web/src/pixel-token.ts`: 시드 문자열(닉네임 또는 라벨) → 해시 → CLAUDE.md §6 팔레트 중
하나의 링 컬러, 라벨 첫 글자를 원 안에 그린다. 같은 시드는 항상 같은 색(결정적) — 재접속해도,
다른 브라우저에서도 같은 토큰이 같은 색으로 보인다. 프리셋 24종 스프라이트는 v1.0 몫이라
손대지 않았다.

### PixiJS 캔버스 (`apps/web/src/pixi/TableCanvas.tsx`)

레이어 순서: 지도 스프라이트 → 그리드 오버레이(`Graphics`) → 토큰(`Container` per 토큰) → 핑
이펙트. 좌표계는 그리드 셀 좌표(PROTOCOL.md대로 소수 허용) ↔ 픽셀 좌표를
`offsetX/offsetY/cellSize`로 변환하는 두 순수 함수(`toPixel`/`toGrid`)로만 다룬다.

드래그는 Pixi 자체 포인터 이벤트로 처리하고, **놓았을 때만** `onTokenDragEnd`로 부모에 알려서
`token.move`를 보낸다(드래그 중에는 React 리렌더를 유발하지 않는다 — 60fps 예산 §8을 지키기
위해). 서버가 거부(`error`)하면 React 쪽에서 `resyncKey`를 증가시켜 캔버스가 `tokens` prop
그대로(마지막으로 승인된 위치) 다시 그리게 만든다 — "조용히 수렴"(§6)을 드래그 실패 케이스까지
확장한 것.

**실제로 겪은 함정 — 드래그가 항상 (0,0)으로 보고되던 버그**: 초기 구현은
`app.stage.on("pointermove", onMove)`로 드래그 추적을 했는데, `app.stage`에 `hitArea`를
지정하지 않으면 Pixi는 자식 도형의 실제 지오메트리(토큰 원, 그리드 선)로만 히트 테스트를
한다 — 포인터가 토큰의 14px 반지름 원 밖으로 나가는 순간 `stage`가 그 지점에서 "히트"되지
않으므로 `pointermove` 콜백이 새 좌표를 받지 못하고, 결과적으로 어디로 드래그하든 항상
드롭 시점의 `token.move`가 시작 좌표(0,0) 그대로 전송됐다(브라우저로 직접 조작하며 발견 —
WS 스파이 클라이언트로 실제 전송된 payload를 찍어봐서 확인). 고전적인 Pixi 함정으로,
공식 드래그 예제가 권장하는 대로 초기화 시 `app.stage.eventMode = "static"`과
`app.stage.hitArea = app.screen`을 한 번 설정해 해결했다(`app.screen`은 렌더러가 리사이즈할
때 갱신되는 참조라 별도 리사이즈 훅이 필요 없다). 이 버그는 `pnpm test`로는 잡히지 않는다 —
반드시 실제 포인터 드래그로 확인해야 한다는 걸 보여주는 사례라 남겨둔다.

### `/assets/` 정적 서빙 충돌 — 웹 번들이 아예 안 뜨던 버그

2단계부터 업로드된 지도 이미지를 `/assets/`에서 서빙해왔는데, Vite 빌드 산출물도 기본적으로
`dist/assets/*.js|css`로 나와서 같은 URL 프리픽스를 두 static 플러그인이 나눠 갖는 충돌이
있었다(운영/Docker처럼 `apps/web`을 빌드해서 서버가 정적으로 서빙하는 모드에서만 드러난다 —
`vite dev`로 개발할 땐 Vite 자체 서버가 번들을 서빙하니 이 충돌이 없다). find-my-way 라우터가
지도-업로드용 정적 플러그인 쪽으로 매칭시켜버려서 웹앱 JS/CSS가 전부 404, 브라우저엔 흰
화면만 떴다. 지도 업로드 정적 서빙을 `/uploads/`로 옮겨 충돌을 없앴다(`apps/server/src/app.ts`,
`routes/tables.ts`의 `publicPath`). **빌드된 dist를 실제로 서빙하며 브라우저로 열어보기
전까지는 안 드러나는 버그**였다 — `pnpm --filter @hearthside/web build`가 성공한다고 해서
서버가 그 산출물을 올바르게 서빙한다는 보장은 없다는 교훈.

### 두 브라우저 탭으로 직접 확인 (이번 세션에서 추가로 수행)

Claude Code의 브라우저 자동화로 실제 서버(`pnpm --filter @hearthside/web build` 산출물 +
`tsx apps/server/src/index.ts`)를 띄우고 두 탭으로 직접 조작해 위 두 버그를 발견·수정한 뒤
재검증했다. 확인한 것: 로그인 → 방 생성 → 초대 링크로 게스트 입장(같은 브라우저 프로필은
쿠키를 탭끼리 공유하므로, 세션 쿠키를 프로그래밍적으로 바꿔가며 DM/플레이어 두 역할을
순차적으로 재현) → DM 전용 패널(지도 업로드/그리드 슬라이더/토큰 추가) 노출 제어 →
토큰 추가 → **토큰 드래그가 다른 탭에도 즉시 반영**(버그 수정 후) → 더블클릭 핑 렌더링 →
`/roll 1d20+5` 굴림과 일반 채팅이 양쪽 탭 모험 일지에 동일하게 표시 → 페이지 새로고침 후
토큰 위치·로그가 스냅샷으로 그대로 복원. 아래 DoD 체크리스트를 갱신했다.

## 시뮬레이터 측정치 (`scripts/sim-clients.ts`, 실제 실행)

로컬(Windows, 개발 머신)에서 `pnpm exec tsx scripts/sim-clients.ts` 직접 실행 — WS 클라이언트
6개(DM 1 + 플레이어 5)가 각자 자기 토큰을 60초 동안 초당 5회 `token.move`로 옮기고, 매번 서버가
그 이동을 되돌려줄 때까지의 왕복 지연을 측정했다(서버는 시뮬레이터와 같은 프로세스에서
`buildApp()`으로 기동 — `table-ws.test.ts`와 동일한 방식):

```
클라이언트 수: 6, 측정 시간: 60.2s, 총 왕복 샘플: 1790
릴레이 왕복 지연 — p50: 1.04ms, p95: 1.75ms, max: 2.77ms
서버 프로세스(시뮬레이터 포함) — 평균 CPU: 2.1%, RSS 종료값: 82.6MB, RSS 피크: 97.0MB
```

목표(p95 < 50ms, 로컬)를 여유 있게 통과했다(1.75ms). CPU/RSS도 §8 예산(CPU < 10%, RSS < 500MB)
안에 든다 — 다만 이건 개발 머신 기준이고, 서버·시뮬레이터가 같은 프로세스라 순수 서버만의
수치보다 다소 보수적(더 낮게 나올 여지가 있다)이다. NAS(J4125)는 이보다 느리므로 실제 배포
후 재측정을 권장한다(2단계에서도 동일하게 권고했던 사항).

  발견해 수정했다 — `pnpm test`와 curl 스모크만으로는 둘 다 잡히지 않았을 결함이다.

## 이번 단계에서 내린 판단 콜 (검토 요청)

- **`GET /api/session` 신설**: 명세에 없던 엔드포인트지만, 테이블 화면이 "나는 누구인가"를
  알아야 하는 요건(자기 토큰 드래그 판단, 참가자 강조)이 있어 최소 형태로 추가했다. 기존
  인증 흐름은 그대로고 조회만 추가한 것이라 리스크가 낮다고 판단했다.
- **SPA `setNotFoundHandler` 폴백**: 명세엔 명시되지 않았지만 "초대 링크는 진짜 URL이어야
  한다"는 요건상 필수라고 판단해 추가했다. `/api`·`/ws`·`/content`·`/assets`는 폴백에서
  제외해 기존 404 계약을 안 건드리게 했다.
- **그리드 캘리브레이션 UI**: 드래그가 아니라 슬라이더 + "저장" 버튼으로 구현했다(값 조정 중엔
  전송 안 하고 명시적으로 저장할 때만 `grid.set`을 보낸다) — 슬라이더를 움직일 때마다
  네트워크를 때리지 않으려는 선택. PROMPT-stage3.md가 "드래그 또는 슬라이더"라고 명시했으니
  범위 안이라고 판단했다.
- **토큰 추가/잠금/삭제 UI**: 프로토콜엔 있지만 화면 요구사항에 명시적 언급이 없어서, DM
  패널에 최소한의 폼(라벨 + 소유자 선택)과 토큰별 잠금 체크박스/삭제 버튼을 붙였다 — 이 UI가
  없으면 애초에 테이블에 토큰이 하나도 안 생겨서 나머지 기능(드래그·잠금)을 테스트할 방법이
  없다고 판단했다.
- **채팅창의 `/roll` 처리**: DoD의 "`/roll 1d20+5 adv` 결과가 양쪽 일지에" 문구를 그대로
  구현 — 채팅 입력창에 `/roll `로 시작하면 `chat.say` 대신 `dice.roll`을 보낸다. DM에게만
  보이는 "비밀" 체크박스를 곁들였다.
