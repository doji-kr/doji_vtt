# PROMPT — 5단계: 스튜디오 v1 (문서 우선 에디터 · 발행 파이프라인)

> 사용법: 리포 루트에서 Claude Code 새 세션에 이 파일 내용을 그대로 전달한다.
> 전제: 4단계 §3(수동 안개)까지 완료된 뒤 시작한다. §4(WebRTC 음성)는 best-effort라
> 미완이어도 이 단계는 막히지 않는다 — 스튜디오는 라이브 테이블과 독립이다.
> 시작 시점의 `pnpm test` 전체 green 개수를 기준선으로 STAGE5.md에 기록하고, 그 기준선은
> 이 단계가 끝나도 전부 green이어야 한다.

## 역할과 컨텍스트

너는 「화롯가」의 리드 엔지니어다. 시작 전에 정독하라: `CLAUDE.md`(특히 **§1.3 채널 분리,
§1.4 실패도 전진, §1.7 문서 우선 에디터**, §5, §6, §8), `packages/schema/SCHEMA.md`(R1~R7),
`docs/STAGE4.md`의 계정 계층(소유권 판단은 오직 `userId` — 닉네임 절대 금지),
`apps/server/src/module-registry.ts`(부팅 스캔 구조 — 이번에 개조 대상이다).

**기존 공개 계약은 변경 금지, 추가만 허용**: schema/runtime 타입·함수, 기존 REST/WS 계약,
기존 테스트 전부. 스키마 자체(`module.json` v0.1)는 이번 단계에서 바꾸지 않는다 —
에디터는 기존 포맷의 *저작 도구*이지 포맷 개정이 아니다.

## 목표

**MVP 문장의 마지막 조각.** 회원(작가)이 브라우저에서 시나리오를 쓰고 「발행」을 누르면
서가에 포스터로 꽂히고, 다른 회원과 게스트가 그것을 혼자 완주한다. 이 단계가 끝나면
원소스 멀티플레이의 "소스"가 파일 폴더가 아니라 사람에게서 나오기 시작한다.

## 핵심 아키텍처 결정

1. **에디터는 문서다, 그래프가 아니다 (§1.7).** 블록 스택 에디터를 React로 자작한다.
   ProseMirror/TipTap/Slate/Lexical 등 리치텍스트 프레임워크와 DnD 라이브러리는 **금지** —
   v1의 본문은 plain textarea + 구조화 블록 폼이면 충분하고, 블록 재정렬은 ▲▼ 버튼이다.
   조감도(그래프 시각화)는 만들지 않는다 — 우측 패널의 "씬별 나가는 연결" 텍스트 목록이
   조감도 v1이다(SVG 그래프는 스트레치, 필수 아님).
2. **초안과 발행본은 다른 컬럼이다.** `scenarios` 테이블에 `draft_json`(항상 존재)과
   `published_json`(발행 시 복사)을 분리한다. 서가·솔로 러너·레지스트리는 **published_json만**
   본다. 초안 저장은 2~3초 디바운스 자동 저장, last-write-wins(단일 작가 전제), 이력 없음
   (버전 이력은 백로그로 STAGE5.md에 명시).
3. **id 네임스페이스.** 파일 모듈은 폴더명 그대로, 스튜디오 발행물은 `st-` 접두 + 짧은
   uuid(예: `st-3f9a1c2b`). `plays.module_id`는 이 네임스페이스를 그대로 참조한다 —
   파일 모듈과의 충돌이 원천적으로 불가능해야 한다.
4. **발행은 린트 게이트다.** `publish` 시 서버가 `lint()`를 실행해 **error 0만 통과**시킨다
   (warn은 통과하되 결과에 동봉). `soloPlayable`(R6)은 배지일 뿐 발행 조건이 아니다 —
   soft edge 위주의 "라이브 전용" 시나리오도 발행할 수 있고, 서가에 라이브 전용으로 표시하되
   솔로 시작 버튼은 `soloPlayable`일 때만 활성화한다. 레지스트리는 "파일 모듈 ∪ DB 발행본"
   병합 뷰로 개조한다 — 부팅 스캔 + publish/unpublish 시점의 갱신 함수 호출이면 충분하다
   (파일 감시/핫리로드 프레임워크 금지).
5. **에디터 API는 채널 분리의 유일한 명시적 예외다.** `dm_notes`를 포함한 module 원문을
   반환하는 API는 `/api/scenarios/*` 군이 유일하며, `owner_user_id === request.userId`가
   아니면 무조건 403이다(다른 회원도, 게스트도). 그 외 모든 API의 dm_notes 부재를 검사하는
   기존 채널 분리 회귀 테스트에 **스튜디오 발행물 케이스를 추가**한다 — 파일 모듈만 검사하던
   테스트가 이제 DB 발행물도 커버해야 한다.
6. **재발행과 진행 중 플레이 — stale 정책.** publish마다 `published_json`의 해시를 저장하고,
   play 생성 시점의 해시를 play에 기록한다. 이어하기 시 해시가 불일치하면 그 플레이는
   stale — "이야기가 개정되어 처음부터 다시 시작합니다" 안내 후 새 플레이를 만든다(옛 로그는
   재생하지 않는다 — 개정된 모듈 위에서의 replay는 깨질 수 있고, 깨진 replay를 어물쩍
   재생하는 것보다 정직한 리셋이 낫다). 발행본 버전 스냅샷 보관은 백로그.

## 산출물 (전부 필수)

### 0. §1 리뷰 후속 — plays의 회원 귀속

`plays`에 `owner_user_id TEXT NULL REFERENCES users(id)` 컬럼과 인덱스를 추가한다.
회원이 만드는 플레이는 `userId`를 기록하고, **목록 조회는 회원이면 `owner_user_id` 기준,
게스트면 기존 nickname 기준**을 유지한다(게스트 솔로 플레이는 계속 되어야 한다 — §1의
판단을 뒤집지 않는다). 이것으로 "동명의 게스트 둘이 서로의 세이브를 본다" 문제가 회원에
한해 해소된다. 기존 행은 NULL 그대로 둔다. 회귀: 게스트 플레이 흐름 테스트가 계속 통과할 것.

### 1. `apps/server` — 스튜디오 API (모두 회원 전용, zod 검증)

| 메서드 | 경로 | 내용 |
|---|---|---|
| POST | `/api/scenarios` | 새 초안 생성 — 3막 원샷 템플릿(산출물 4)으로 시딩, 201 + 원문 |
| GET | `/api/scenarios` | 내 서재: id, 제목, 상태(draft/published), updated_at, 최근 lint 요약 |
| GET | `/api/scenarios/:id` | 원문(draft_json). **owner만, 아니면 403** |
| PUT | `/api/scenarios/:id` | draft 저장. 페이로드는 `parseModule` 통과 필수(파스 불가면 400 — 항상 파스 가능한 초안만 존재한다). 응답에 lint 결과 동봉 |
| POST | `/api/scenarios/:id/publish` | lint error 0 게이트 → `published_json`/해시/`published_at` 갱신, 레지스트리 갱신 |
| POST | `/api/scenarios/:id/unpublish` | 서가에서 내림(레지스트리 갱신). 진행 중 play는 이어하기 시점에 "이야기가 서가에서 내려갔습니다" 안내 |
| DELETE | `/api/scenarios/:id` | draft만 즉시 삭제. published 상태면 unpublish 선행을 요구(409) |
| POST | `/api/scenarios/:id/preview-plays` | **draft_json 기준** 프리뷰 플레이 생성(owner 전용) |

- `plays`에 `is_preview INTEGER DEFAULT 0` 추가 — 프리뷰 플레이는 서가 "이어하기" 목록과
  일반 plays 조회에 절대 노출되지 않는다.
- 파스 실패(400)와 린트 경고는 **다른 층**이다 — 저장은 파스만 요구하고, 발행이 린트를 요구한다.

### 2. 레지스트리 병합 + 서가 확장

`loadModuleRegistry`를 "파일 ∪ DB published" 병합 뷰로 개조. `ModuleSummary`에
`author_display_name`(파일 모듈은 "화롯가 내장"), `published_at`, 라이브 전용 여부를 추가.
서가 카드에 글쓴이·라이브 전용 표시, 솔로 시작 버튼은 soloPlayable만 활성.

### 3. `apps/web` — `/studio` (이야기꾼의 책상)

- **서재 목록**: 내 시나리오 카드(상태 뱃지 draft/published, lint 요약, 최근 수정), "새 이야기" 버튼.
- **편집 화면(3패널, §6 미학 — 중앙은 양피지 책상)**:
  - 좌: 씬 목록(추가/삭제/▲▼), 메타(제목·로그라인·난이도·예상시간·태그·시작 씬)·NPC(3줄)·
    플래그 선언 서랍.
  - 중앙: 선택된 씬 편집 — `read_aloud`/`dm_notes` 영역(라벨로 채널을 명확히: "플레이어가
    듣는 것" / "나만 아는 것"), `secrets` 편집, 블록 스택(판정·선택지·조우·핸드아웃·비밀
    삽입 메뉴, 각 블록은 타입별 폼, `goto`는 존재 씬 드롭다운 + 「+ 새 씬」 즉석 생성,
    check는 on_success/on_fail 양쪽 폼 — 스키마가 이미 fail 분기를 강제하니 UI도 그렇게).
  - 우: 린터 패널(저장마다 자동 실행, R1~R7을 error/warn별로 sceneId·hint와 함께, 클릭 시
    해당 씬으로 점프), soloPlayable 배지 미리보기, 「프리뷰」(= 기존 솔로 러너 화면을
    프리뷰 플레이로 그대로 재사용) · 「발행」 버튼.
- 자동 저장 상태 표시("잉크가 마르는 중… / 저장됨"), 발행 성공 시 서가로 이동 + 포스터 확인.

### 4. 템플릿 — "3막 원샷" 스켈레톤

씬 5개(훅 / 전개 2 / 절정 / 에필로그·엔딩), read_aloud에 작성 안내 TODO 문장, 최소 hard
edge로 연결되어 **생성 직후부터 lint error 0**인 상태로 시작한다 — 빈 페이지가 아니라
"이미 이어져 있는 뼈대"를 준다.

### 5. 테스트

- 서버 e2e: 초안 생성 → 편집(PUT) → 경고 있는 상태에서 publish 시도(경고는 통과) /
  error 있는 초안 publish 거부 → 발행 → 서가 노출 → **타 회원 GET/PUT 403 · 게스트 403** →
  발행물로 신규 play 완주 → 재발행(내용 변경) → 기존 미완료 play 이어하기 시 stale 안내 →
  unpublish → 서가 소멸.
- 프리뷰 격리: preview play가 이어하기·목록 어디에도 노출되지 않음.
- 채널 분리 회귀 확장: **DB 발행물**의 dm_notes/Secret.dm_notes/Npc.secret 문자열이
  서가·플레이 API 어디에도 없음.
- plays 회원 귀속: 같은 닉네임의 게스트 둘의 목록이 섞이던 케이스가 회원에겐 발생하지 않음
  + 게스트 흐름 무회귀.

### 6. `docs/STAGE5.md` + README

작업 기록 관례 유지(결정·판단 콜·검증·핸드오프). README 로드맵 표에 5단계 행
("스튜디오 v1 — 문서 우선 에디터·발행 파이프라인")을 추가하고 완료 시 ✅. 4단계 행은
기존 관례대로 §3·§4 완료 시점에 정리한다(이 단계에서 건드리지 않는다).
