# PROMPT — 3단계: 라이브 테이블 코어 (서버 권위 WS 릴레이 + PixiJS 지도·토큰 + 실시간 주사위)

> 사용법: 리포 루트에서 Claude Code 새 세션에 이 파일 내용을 그대로 전달한다.
> 전제: 1·2단계 완료 — 테스트 33개 green, 서버 권위 솔로 러너가 Docker로 NAS에 올라가 있다.

## 재범위 공지

2단계 예고에서 3단계에 묶여 있던 **WebRTC 음성 + coturn, 계정 본편(argon2id)** 은 4단계로
옮긴다. 이유: "한 단계, 한 리스크." 3단계의 리스크 코어는 **서버 권위 실시간 릴레이와 캔버스
동기화**이고, WebRTC NAT 통과는 성질이 전혀 다른 별개의 리스크다. 이번 세션 동안 음성은
디스코드를 병행하면 된다 — 테이블이 먼저다.

## 역할과 컨텍스트

너는 「화롯가」의 리드 엔지니어다. 시작 전에 정독하라: `CLAUDE.md`(특히 §1, §5, §7 이벤트
명명, §8 성능 예산), `docs/STAGE2.md`(핸드오프 항목), `packages/schema/SCHEMA.md`.

**기존 공개 계약은 변경 금지, 추가만 허용이다**: schema/runtime의 기존 타입·함수 시그니처,
2단계 REST API는 그대로 둔다. R7 린터 규칙은 "추가"이므로 허용된다. 기존 테스트 33개는
이 단계가 끝나도 전부 green이어야 한다(솔로 러너 회귀 금지).

## 목표

DM이 방을 열고, 지도를 깔고, 토큰을 놓고, 친구가 초대 링크로 들어와, **같은 지도 위에서
토큰이 움직이고 주사위가 구른다.** 브라우저 두 개로 이 전부가 되면 3단계는 끝난 것이다.

## 핵심 아키텍처 결정

1. **실시간도 서버 권위다.** 클라이언트는 op를 제안하고, 서버가 검증·순서 부여(seq)·적용 후
   브로드캐스트한다. 낙관적 UI는 **`token.move` 드래그 프리뷰에만** 허용하고, 서버 확정치가
   오면 조용히 수렴한다.
2. **주사위는 서버가 굴린다.** 공정성의 근거는 서버의 crypto RNG 하나뿐이다. 클라이언트가
   굴림 결과를 보내는 경로는 존재하지 않는다. (솔로 러너의 `resolveCheck`는 기존대로 —
   그쪽은 혼자 노는 방이다.)
3. **방 상태는 메모리 + 디바운스 스냅샷(SQLite, 2~5초)** — CLAUDE.md §5. 컨테이너가
   재시작해도 방은 그대로 있다.
4. **DM의 비밀 굴림은 채널 분리의 확장이다.** `visibility: "gm"` 굴림의 결과는 DM 소켓에만
   나간다. 플레이어 채널로 새지 않음을 테스트로 증명한다(2단계 채널 분리 테스트의 실시간판).

## 산출물 (전부 필수)

### 1. R7 린터 규칙 — 2단계 핸드오프 처리

`packages/schema` 린터에 추가: **도달 가능한 모든 choice 블록에, `requires_flag`와 무관하게
항상 노출되는 옵션이 최소 1개 있는가.** 없으면 warn(플래그 조합상 도달 시점에 항상 참일 수도
있으므로 error는 과하다 — 근거를 SCHEMA.md에 쓴다). 위반/통과 픽스처, SCHEMA.md 린터 표 갱신.

### 2. `docs/PROTOCOL.md` — 실시간 프로토콜 v0

- 봉투: `{ seq, room_id, actor, type, payload }` — 모든 payload는 zod 스키마.
- c2s ops / s2c events 표. 이벤트 명명은 §7(`도메인.동사`):
  `table.join` `token.add` `token.move` `token.remove` `map.set` `grid.set`
  `dice.roll` `chat.say` `ping.place` 최소 세트.
- 재접속 절차: `hello(last_seq?)` → `state.snapshot{state, seq}` → 이후 이벤트 스트림.
- 권한 표: DM(방 생성자) = 지도/그리드 설정, 토큰 추가·삭제·잠금, 비밀 굴림.
  플레이어 = 자기 토큰 이동, 굴림, 채팅, 핑. 위반 op는 에러 응답(연결 유지).
- 하트비트/타임아웃, 순서 보장(seq 단조 증가) 규칙.

### 3. `apps/server` — 실시간 + 방

- WS 엔드포인트(`@fastify/websocket` — 채택 근거를 STAGE3.md에 한 줄). 기존 쿠키 세션으로 인증.
- SQLite: `tables(id, name, owner_nickname, invite_token, map_path, grid_json,
  state_json, updated_at, ...)` — `state_json`이 디바운스 스냅샷이다.
- REST 추가: 방 생성/목록/초대 링크(`/t/:invite_token`으로 게스트 입장 — 초대코드+닉네임
  기존 플로우 재사용), 지도 업로드 `POST /api/tables/:id/map`
  (png/jpg/webp 화이트리스트, 8MB 제한, `/data/assets/` 저장, 경로 탈출 방지).
- 주사위 파서: `1d20+5`, `2d6-1`, `adv`/`dis`, `gm`(비밀). 파서는 순수 함수 + 단위 테스트.
  굴림 결과는 방 로그에 저장(재접속 시 일지 복원).

### 4. `apps/web` — `/table/:id` 화면 (PixiJS v8은 여기서 처음 도입)

- 캔버스 레이어: 지도 스프라이트 → 그리드 오버레이 → 토큰 → 핑 이펙트.
- 그리드 캘리브레이션 UI: 셀 크기·오프셋을 드래그/슬라이더로 맞추고 저장(DM 전용).
- 토큰 v0 = **제너레이티브 픽셀 토큰**(닉네임 이니셜 + §6 팔레트 링 컬러, 결정적 시드).
  프리셋 24종 스프라이트는 v1.0 에셋 작업이다 — 지금 그리지 마라.
- 드래그 스냅 이동(낙관적 프리뷰 + 서버 수렴), DM 잠금 토큰은 플레이어가 못 움직인다.
- 더블클릭 핑(링 파동, §6 촛불 앰버).
- 우측 "모험 일지": 채팅(전체/귓속말) + 굴림 카드 통합, 자연 20 금빛 강조(§6),
  `gm` 굴림은 DM에게만 보인다. 참가자 목록(접속 상태 점).
- 미학은 §6 — 숯 배경, 지도만 빛나게. 세션 화면 목업(`mockups/hwarotga-session-mockup.svg`)이
  방향이다. 이번 단계 범위는 그중 지도·토큰·일지·핑까지(안개·이니셔티브 리본·트레이 고도화는 4단계).

### 5. `scripts/sim-clients.ts` — 부하 시뮬레이터

WS 클라이언트 6개가 60초간 초당 수 회 `token.move`를 보내는 시뮬레이션 →
릴레이 지연 p50/p95와 서버 CPU/RSS를 출력하고 STAGE3.md에 기록. 로컬 기준 p95 < 50ms 목표.

### 6. 테스트

- WS e2e: 두 클라 동기화(순서·seq), 재접속 스냅샷 정합, 권한 위반 거부, 게스트 초대 링크 입장.
- **비밀 굴림 채널 분리 테스트**: `gm` 굴림 결과 문자열이 플레이어 소켓 수신분 어디에도 없다.
- 주사위 파서 단위 테스트(경계: `0d6`, `1d20+`, adv+gm 조합 등).
- 기존 33개 포함 전부 green.

### 7. `docs/STAGE3.md` — 작업 기록 (관례)

결정과 근거 / 시뮬레이터 측정치 / 미결·4단계 핸드오프.

## 제약

- 새 의존성은 `@fastify/websocket`(또는 ws)과 `pixi.js` 수준. 그 외는 커밋 메시지에 정당화(§3).
- 지도 업로드가 생겼으니 픽셀 아트 원칙(§1.8)을 기억하라 — 압축·리사이즈로 씬 초기 로드 < 5MB(§8).
- 성능 예산 §8 준수. 시뮬레이터 수치가 근거다.

## 진행 방식

1. **먼저 제시하고 멈춰라**: `PROTOCOL.md` 초안(이벤트 표 + 재접속 절차 + 권한 표),
   `tables` DDL, 화면 흐름(서가 → 방 만들기 → 초대 → 테이블) 텍스트 다이어그램. 승인 후 구현.
2. 순서: R7 → server(WS+방+주사위, 테스트 병행) → web(PixiJS) → 시뮬레이터 → docs.
3. 단계마다 실행 증거(테스트 출력, 시뮬레이터 수치, 스크린샷 가능하면). 보고는 한국어.
